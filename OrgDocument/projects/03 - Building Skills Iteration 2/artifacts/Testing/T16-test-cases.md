## Overview

Performance benchmarks for hierarchical diagram generation and VS Code rendering. All tests
validate that the EDPS skill workflow meets the ≤30 second generation time requirement (TC-303),
proper Mermaid rendering of box-syntax diagrams (TC-301, TC-302), and acceptable performance at
5+ hierarchy levels.

**Measurement approach**: Since EDPS skills run as GitHub Copilot AI conversations,
"generation time" means elapsed wall-clock time from skill invocation to fully formed output
(collaboration.md + main.md + process.md + domain-model.md). Measurement is conducted with a
stopwatch during live test execution; representative medians across three runs are recorded.

## Category Perf-L: Level-Based Generation Time Benchmarks

> Verifies FR-T16.1 — generation time stays within 30-second budget at every hierarchy level.

### TC-T16-001: Level 1 — Single Decomposition Benchmark

**Scenario**: Decompose one control-type participant from a root-level collaboration diagram.  
**Input**: 5-participant diagram, 8-message sequence, 1 control-type target.  
**Expected Result**: Full sub-process folder (4 files) generated in ≤30 seconds.  
**Pass Criterion**: ≤30 s elapsed; all 4 files (`collaboration.md`, `main.md`, `process.md`, `domain-model.md`) present and valid.

---

### TC-T16-002: Level 2 — Two-Level Cascade Benchmark

**Scenario**: Decompose a Level 0 control participant, then immediately decompose one control participant within the resulting Level 1 diagram.  
**Input**: Root: 6 participants, 10 messages. Level 1: 4 participants, 7 messages.  
**Expected Result**: Both decompositions complete; total elapsed ≤60 s (≤30 s each step).  
**Pass Criterion**: Nested sub-folder structure `[root]/01-[A]Boundary/01-[B]Boundary/` created; hierarchy-metadata.json depth correctly recorded as 2.

---

### TC-T16-003: Level 3 — Three-Level Cascade Benchmark

**Scenario**: Chain three successive decompositions: Level 0 → Level 1 → Level 2.  
**Input**: Each level diagram has 5 participants and 8 messages.  
**Expected Result**: Three nested sub-process folders; each step ≤30 s.  
**Pass Criterion**: hierarchy-metadata.json reports `max_depth: 3`; all navigation breadcrumbs correct; total time ≤90 s.

---

### TC-T16-004: Level 4 — Four-Level Cascade Benchmark

**Scenario**: Chain four successive decompositions.  
**Input**: Each level: 4 participants, 6 messages.  
**Expected Result**: Four-level nesting; each decomposition ≤30 s; scale management warns at Level 4 if complexity thresholds exceeded.  
**Pass Criterion**: `max_depth: 4`; no performance degradation observed vs. Level 1–3 benchmarks (within 20% variance).

---

### TC-T16-005: Level 5 — Five-Level Cascade Benchmark (Boundary Condition)

**Scenario**: Chain five successive decompositions reaching the project's minimum supported depth.  
**Input**: Each level: 4 participants, 5 messages.  
**Expected Result**: Five-level nesting without any generation failure or timeout.  
**Pass Criterion**: `max_depth: 5`; all 20 generated files (4 per level) valid; each step ≤30 s; `hierarchy-index.md` Mermaid flowchart includes all 5 levels.

---

### TC-T16-006: Broad Level 1 — Single Level with 8 Parallel Decompositions

**Scenario**: Decompose 8 different control-type participants from one root diagram in sequence.  
**Input**: Root diagram with 10 participants (8 control-type, 2 entity), 15 messages.  
**Expected Result**: 8 sub-folders created; each decomposition ≤30 s.  
**Pass Criterion**: No naming collisions (collision-resolution §2b activates if needed); folder-creation.log has 8 entries; hierarchy-metadata.json breadth at Level 1 = 8.

---

### TC-T16-007: Large Diagram Benchmark — 12-Participant Root

**Scenario**: Root diagram at the scale-management `critical` threshold (12 participants) is decomposed.  
**Input**: 12 participants, 20 messages, 3 control-type targets.  
**Expected Result**: Scale management advisory fires; decomposition still completes ≤30 s.  
**Pass Criterion**: `complexity_metrics.warning_level = "advisory"` or `"critical"` in hierarchy-metadata.json; generation completes within time budget.

## Category Perf-V: VS Code Rendering Validation

> Verifies FR-T16.2 and TC-302 — Mermaid diagrams render correctly in VS Code preview.

### TC-T16-008: Box Syntax Renders in VS Code Preview

**Scenario**: Open a collaboration.md generated by `hierarchy-management` in VS Code Mermaid preview.  
**Input**: `collaboration.md` containing `box` syntax with two named boundaries and a participant per boundary.  
**Expected Result**: Preview shows bounded regions; no syntax errors; participants correctly positioned inside boxes.  
**Pass Criterion**: VS Code Mermaid preview (Markdown Preview Enhanced or built-in) shows diagram without error panel.

---

### TC-T16-009: Nested Sub-Process Navigation Links Render

**Scenario**: Open `main.md` at Level 2 in VS Code; verify breadcrumb and sub-process links render as clickable markdown links.  
**Input**: `main.md` with breadcrumb `> Root > Level 1 > Level 2`.  
**Expected Result**: All relative path links resolve within VS Code workspace; links clickable in rendered preview.  
**Pass Criterion**: No broken-link indicators in VS Code markdown preview; three breadcrumb levels visible.

---

### TC-T16-010: Hierarchy Index Flowchart Renders

**Scenario**: Open `hierarchy-index.md` in VS Code preview after a five-level hierarchy is created.  
**Input**: `hierarchy-index.md` with Mermaid `flowchart TD` of 5 nodes and `click` directives.  
**Expected Result**: Flowchart renders with all nodes and directional arrows; click directives do not cause syntax errors.  
**Pass Criterion**: Diagram visible in preview; no error banner; all 5 nodes labeled correctly.

---

### TC-T16-011: Process Flowchart Renders at Level 3

**Scenario**: Open `process.md` for a Level 3 sub-process in VS Code preview.  
**Input**: `process.md` containing a `flowchart TD` generated by documentation-automation.  
**Expected Result**: Flowchart fully visible; no truncation or rendering failure.  
**Pass Criterion**: All nodes and edges render; VS Code doesn't time out loading the preview (<5 s).

### TC-T16-012: Domain-Model Class Diagram Renders at Level 2

**Scenario**: Open `domain-model.md` for a Level 2 sub-process containing a `classDiagram` block.  
**Input**: 6-class diagram with 8 relationships.  
**Expected Result**: Class diagram renders completely; no missing nodes.  
**Pass Criterion**: All 6 classes visible in preview; relationship arrows drawn correctly.

### TC-T16-013: Large Hierarchy-Index Flowchart (8 nodes) Renders Without Timeout

**Input**: `hierarchy-index.md` with 9-node flowchart (root + 8 children).

### TC-T16-014: EDPS Compliance Report Mermaid Section Renders

## Category Perf-B: Bottleneck Identification

> Verifies FR-T16.3 — skill bottlenecks identified and optimization guidance documented.

### TC-T16-015: Identify Slowest Step in Hierarchy Decomposition Workflow

**Scenario**: Measure time spent in each of the 7 workflow steps of hierarchy-management during a Level 1 decomposition.  
**Input**: Standard 5-participant, 8-message diagram.  
**Expected Result**: Steps 3, 4, 4b, 4c (file generation) account for the majority of elapsed time; Steps 1, 2, 6 are fast validation/metadata operations.  
**Pass Criterion**: A per-step timing breakdown is documented; file generation steps cumulatively ≤25 s; metadata/validation steps cumulatively ≤5 s.

---

### TC-T16-016: Link-Integrity Check Scales with Hierarchy Depth

**Scenario**: Run link-integrity check (§7c) on a 5-level hierarchy with 20 total nodes.  
**Input**: `hierarchy-index.md` pointing to 20 `main.md` files.  
**Expected Result**: Link verification completes in ≤30 s regardless of tree size.  
**Pass Criterion**: All links resolved; `folder-creation.log` appended; time ≤30 s for 20-node tree.

---

### TC-T16-017: Scale Management Report Generation Does Not Delay Decomposition

**Scenario**: Decompose a complexity-threshold-triggering diagram (12 participants) and verify scale analysis does not push generation beyond 30 s.  
**Input**: 12-participant diagram; scale management triggers at §6.  
**Expected Result**: Scale analysis runs and appends to hierarchy-metadata.json without delaying the main decomposition output.  
**Pass Criterion**: Decomposition output files present ≤30 s from invocation; scale analysis report appended thereafter.

---

### TC-T16-018: documentation-automation Content Guard Pre-Check Does Not Add Excessive Overhead

**Scenario**: Invoke documentation-automation on a stub file (marker present) and a fully written file (no marker); compare time overhead of Content Guard.  
**Input (a)**: Stub file with `[TO BE GENERATED - invoke documentation-automation]` marker.  
**Input (b)**: Fully written `collaboration.md` (no marker).  
**Expected Result**: Content Guard adds ≤2 s overhead in both cases (line-count threshold check).  
**Pass Criterion**: Time difference between (a) and (b) invocations ≤2 s; no user-blocking prompt for stub files in `--force` mode.
