## Description

During architectural review, `documentation-automation` and `orgmodel-update` were found to write the same orgModel files (`main.md`, `process.md`, `collaboration.md`, `domain-model.md`) from different input sources with no documented sequencing contract. While they serve complementary purposes (different trigger points, different primary inputs), running `orgmodel-update` after `documentation-automation` risks overwriting hierarchy-aware breadcrumb navigation and box-syntax collaboration diagrams with older non-hierarchical versions.

This task establishes a pipeline ordering contract between the two skills and adds a guard in `orgmodel-update` to detect and defer to `documentation-automation` for folders that are managed by an EDPS hierarchy.

## Objectives

- Document the authoritative pipeline ordering: `documentation-automation` runs after hierarchy decomposition and owns process-level files; `orgmodel-update` operates at the project analysis level
- Add an EDPS-HIERARCHY detection guard to `orgmodel-update`
- Ensure both SKILL.md files cross-reference each other's scope

## Detailed Requirements

### Functional Requirements

- **FR-T22.1**: Add a **Scope and Pipeline Position** section to both `documentation-automation/SKILL.md` and `orgmodel-update/SKILL.md` explaining their relative roles:
  - `documentation-automation`: triggered by hierarchy decomposition events; input is `collaboration.md`; owns process-level files at each EDPS hierarchy node.
  - `orgmodel-update`: triggered by project analysis (domain alignment, concept extraction); input is `domain-alignment.json` / `domain-concepts.json`; operates at the org model process folder level (not per-hierarchy-node).
  - Ordering rule: within a single process, `documentation-automation` always runs after any `hierarchy-management` decomposition. `orgmodel-update` runs from project analysis outputs. If both apply to the same folder, `documentation-automation` takes precedence for the four shared files.

- **FR-T22.2**: Add an **EDPS-Hierarchy Guard** to `orgmodel-update/SKILL.md` Workflow, as the first step before any file generation:
  1. For each target orgModel folder, check if `hierarchy-metadata.json` exists in the folder or any of its ancestor folders within `orgModel/`.
  2. If `hierarchy-metadata.json` is found: the folder is EDPS-hierarchy-managed. For the four shared files (`main.md`, `process.md`, `collaboration.md`, `domain-model.md`), `orgmodel-update` must **skip direct generation** and instead append its analysis outputs (new domain entities, new vocabulary terms) to a **pending-updates file** (`[folder]/pending-orgmodel-updates.md`) for the practitioner to manually merge, or log a recommendation to re-run `documentation-automation` with the updated `domain-concepts.json` as context.
  3. If no `hierarchy-metadata.json` found: generate files normally as current behavior.

- **FR-T22.3**: `orgmodel-update` must always update `vocabulary.md` and `test-case-list.md` regardless of EDPS-hierarchy status, as these are not generated by `documentation-automation`.

- **FR-T22.4**: Add a **Related Skills** section to both SKILL.md files listing the other skill and its scope, so practitioners can navigate the pipeline easily.

### Technical Requirements

- **TR-T22.1**: The guard check for `hierarchy-metadata.json` must search the folder itself and up to 3 ancestor levels within `orgModel/` (to handle both root-level and deeply nested folders).
- **TR-T22.2**: The `pending-orgmodel-updates.md` file format must include: source analysis file reference, list of proposed additions (entities, vocabulary terms, process notes), and a human-action prompt.

### Business Requirements

- **BR-T22.1**: A practitioner must not lose hierarchy-aware navigation (breadcrumbs, Sub-Processes tables, box-syntax diagrams) by running `orgmodel-update` on hierarchy-managed folders.
- **BR-T22.2**: Domain analysis outputs (new concepts from Project 1) must still be incorporated into hierarchy-managed orgModel folders — via the pending-updates mechanism rather than silent overwrite.
