## Description

During architectural review, rules VR-1 through VR-4 (boundary diagram validation rules) were found fully duplicated in both `diagram-generatecollaboration/SKILL.md` and `edps-compliance/SKILL.md` (Group A). The `edps-compliance` skill acknowledges these as "inherited from `diagram-generatecollaboration`" but reproduces all check algorithms independently. This creates a single-source-of-truth violation: any future change to a VR rule must be applied in two places and is likely to drift.

This task eliminates the duplication by making `edps-compliance` delegate VR-1–VR-4 checking to `diagram-generatecollaboration` rather than reimplementing it.

## Objectives

- Remove the duplicated VR-1–VR-4 check algorithms from `edps-compliance/SKILL.md`
- Add `boundary-validation-report.json` (output of `diagram-generatecollaboration`) as an optional input to `edps-compliance`
- Define a clear delegation sub-step: if the report is provided, import VR results; if not, invoke the validation mode of `diagram-generatecollaboration` automatically
- Ensure `edps-compliance`'s compliance score correctly incorporates the delegated VR results

## Detailed Requirements

### Functional Requirements

- **FR-T18.1**: `edps-compliance` inputs section must list `boundary-validation-report.json` as an optional input, generated by `diagram-generatecollaboration`.
- **FR-T18.2**: In the `edps-compliance` workflow, Step 2 (currently "Apply Group A Rules") must be replaced with a delegation step:
  1. If `boundary-validation-report.json` is present for the target `collaboration.md`, read VR-1–VR-4 results directly from it.
  2. If not present, invoke the `diagram-generatecollaboration` boundary validation mode on the target file and capture its output.
  3. Map the imported VR results into the `edps-compliance` rule result format (same `id`, `status`, `severity`, `message` fields).
- **FR-T18.3**: Remove the full VR-1–VR-4 check algorithm blocks from `edps-compliance/SKILL.md` Section 2. Retain the rule catalogue table (with descriptions) as documentation, but removed all step-by-step implementation text.
- **FR-T18.4**: The compliance score formula must continue to include VR-1–VR-4 results (either PASS/FAIL/SKIPPED) in its `total_checks` and `passed_checks` counts.
- **FR-T18.5**: The `diagram-generatecollaboration/SKILL.md` remains the single canonical location for all VR rule definitions and check algorithms. No changes required to it for this task.

### Technical Requirements

- **TR-T18.1**: The delegation step must be idempotent: running `edps-compliance` twice on the same target must not regenerate or overwrite a pre-existing `boundary-validation-report.json`.
- **TR-T18.2**: If the delegation invocation fails (e.g., the file is malformed), `edps-compliance` must mark VR-1–VR-4 as `SKIPPED` with reason `delegation-failed` and continue checking Group B and Group C rules.

### Business Requirements

- **BR-T18.1**: A practitioner running both skills on the same diagram must receive consistent VR verdicts — same PASS/FAIL outcome from both.
- **BR-T18.2**: Adding a new VR rule in a future iteration should require changes only in `diagram-generatecollaboration/SKILL.md`.
